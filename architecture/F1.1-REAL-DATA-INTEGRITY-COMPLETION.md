## Executive Summary

In demo mode the app rendered fabricated data that did not survive a refresh
and, for Analytics, changed on every reload. The root cause was that demo-mode
data was generated fresh (some of it randomly) on each read and never
persisted. This sprint introduces a small **localStorage-backed demo
persistence store** seeded once from the existing demo constants, and routes
the demo branches of the data service through it. All six issues are fixed
without touching the backend path, the Storage Gateway, KV/SQL authority, API
contracts, or any layout.

---

## Root Causes & Fixes

1. **Dummy/hardcoded data** ‚Äî Multiple modules generated or hardcoded their own
   demo data. Consolidated onto one persisted demo store; removed the divergent
   copies (see items below).

2. **Dashboard did not display persisted questionnaire answers** ‚Äî In demo mode
   `DiagnosticRoute` never called `createSubmission`, so answers were discarded;
   and the CORTEX detail view always used static mock data.
   Fix: `createSubmission` now runs in demo mode and persists the full
   submission (with answers) to the demo store; `CortexDashboard` builds its
   view from the persisted submission's real answers (`generateCortexData`) when
   present, falling back to mock only when there are none.

3. **Analytics regenerated values on refresh** ‚Äî `AnalyticsDashboard` built 20
   submissions with `Math.random()` scores/dates on every load.
   Fix: removed the random generator; Analytics now reads the persisted demo
   submissions (stable across refreshes) and computes analytics from them.

4. **Platform Health used fabricated counts** ‚Äî `SettingsPage` hardcoded
   `{ new:5, in-review:3, completed:8, approved:2, total:18 }`.
   Fix: counts are derived from the persisted submissions (real data / graceful
   zero-state); the data-service demo `getPlatformSettings` was aligned too.

5. **Email Queue injected fake demo leads** ‚Äî `EmailNurturePanel` called
   `seedDemoQueue()` on every refresh, polluting the real queue.
   Fix: removed the seeding call; the queue now shows only real persisted emails
   (created when a diagnostic is submitted) with a proper empty state.

6. **Invited team members did not persist** ‚Äî demo invite/update/remove only
   mutated local component state.
   Fix: the demo team roster is persisted; invite/update/remove write through to
   the store, so members survive a refresh.

## QA Follow-up ‚Äî MC-0002 (Dashboard did not display persisted diagnostic data)

**Reported:** After a diagnostic submission completes, the expected data did not
appear on the Dashboard.

**Root cause:** The default Dashboard view (Command Center) rendered a **static
hardcoded `ACTIVITY_FEED`** in the "Recent Activity" panel ‚Äî fabricated events
(e.g. "Proposal draft ¬ß3 updated ‚Äî TechCorp Solutions", "New diagnostic
submitted ‚Äî FinanceHub"). It never reflected the persisted submissions, so a
freshly submitted diagnostic could not appear there. (The persisted submission
was already reaching the "All Leads" list and CORTEX detail; the gap was
specifically the default Command Center activity feed.)

**Fix (smallest path):** Replaced the static array with `buildActivityFeed(submissions)`,
which derives the feed from the real persisted submissions (newest first, with a
relative-time label) and renders a proper empty state when there are none. No
layout change; removed the hardcoded events (also advances issue #1).

**File:** `src/app/components/TeamHomeDashboard.tsx`

**Verification:** `vite build` passes; smoke test passes; a UI test confirmed a
submitted diagnostic appears in Recent Activity with a live "just now / min ago"
timestamp and that the old hardcoded strings are gone.

## Supabase Backend Path ‚Äî Root Cause & Fix (QA round 3)

**Trace:** UI action ‚Üí `dataService.*` ‚Üí `isDemo()` gate (`!FEATURES.BACKEND_INTEGRATION`)
‚Üí (demo branch: localStorage)  OR  (live branch: `@/app/lib/api` ‚Üí Supabase edge
function `make-server-324f4fbe` ‚Üí `kv_store_324f4fbe` Postgres table + Supabase Auth).

1. **Deployed value of `BACKEND_INTEGRATION`:** `false`. The flag is
   `envFlag('VITE_BACKEND_INTEGRATION', false)` ‚Äî default `false`, and nothing in
   the repo (no `vercel.json`, only `.env.example` which set it `false`) turns it
   on. So unless the Vercel project explicitly sets `VITE_BACKEND_INTEGRATION=true`,
   the deployed app runs in demo mode.

2. **Why the demo/localStorage branch was used:** `isDemo()` returns
   `!FEATURES.BACKEND_INTEGRATION`, i.e. `true` when the flag is false. Every
   `dataService` function therefore took its demo branch. Additionally the
   default Dashboard (`TeamHomeDashboard`) read `getDemoSubmissions()` /
   `getDemoTeamMembers()` **unconditionally**, so even flipping the flag would
   not have shown real data on that screen.

3. **Supabase tables per record (authority = KV):**
   - Diagnostic submissions ‚Üí `kv_store_324f4fbe` under keys `sub:<id>`,
     `sub_email:<email>`, and the `submissions:index` list (written by
     `POST /submissions`). (Shadow SQL table `public.diagnostic_submissions`
     exists per migrations; KV remains the write authority ‚Äî unchanged.)
   - Team members ‚Üí **Supabase Auth** users (`POST /team/invite` ‚Üí
     `auth.admin.createUser`) plus role metadata in `kv_store_324f4fbe`.
   - Email queue ‚Üí `kv_store_324f4fbe` (`email_queue:*`).
   - Analytics inputs & Platform-health inputs ‚Üí derived server-side from the
     submissions in `kv_store_324f4fbe` (`GET /analytics/overview`, `GET /settings`).

4. **Smallest root-cause fix:**
   - `src/config/features.ts` ‚Äî default `BACKEND_INTEGRATION` to
     `import.meta.env.PROD`. **Production builds (the deployed app) now default to
     the real Supabase backend**; local dev defaults to demo. Explicit
     `VITE_BACKEND_INTEGRATION` still overrides either way.
   - `src/app/components/TeamHomeDashboard.tsx` ‚Äî read submissions and team
     members from `getSubmissions` / `getTeamMembers` (Supabase) when the backend
     is enabled, falling back to the demo store only in demo mode. (Data-source
     change only; layout untouched.)
   - `.env.example` ‚Äî documents that production defaults to the real backend and
     that demo mode is now an explicit opt-out.

5. **Demo mode kept & isolated:** all demo branches remain; demo is now reached
   only in local dev or by explicitly setting `VITE_BACKEND_INTEGRATION=false`.

**Evidence:**
- Dev/demo smoke test (`npm run dev` ‚Üí demo): passes ‚Äî no Supabase calls, demo
  credentials accepted locally.
- Production build probe (`vite build` + `vite preview`): team login fired a real
  request to `POST https://tmclwqcgqfcmqwgrogjy.supabase.co/functions/v1/make-server-324f4fbe/auth/team/login`
  ‚Äî proving the deployed build runs in backend mode (in demo mode no Supabase
  request is made). Because all `dataService` calls share the same flag,
  `createSubmission` / `inviteTeamMember` route to Supabase identically.
- **Live row-in-Supabase evidence (create diagnostic ‚Üí confirm row ‚Üí refresh;
  invite member ‚Üí confirm row) could NOT be captured from this environment:** the
  egress policy blocks `*.supabase.co` (proxy returns 403 CONNECT), and no
  service-role DB credentials are available here. This is an environment
  limitation, not a code gap ‚Äî the network capture above proves the request is
  issued to the real endpoint. This must be confirmed once in the deployed
  environment (see "Deployment note").

**Deployment note:** the code now defaults production to the backend. If the
Vercel project has an explicit `VITE_BACKEND_INTEGRATION=false`, it must be
removed (or set to `true`) for the default to take effect. The Supabase edge
function must be deployed and its secrets set (per `dataService.ts` header) and
team users seeded in Supabase Auth for login.
