## Executive summary

MARQ Cortex V2 today persists all production business data in a single Supabase PostgreSQL table (`kv_store_324f4fbe`) accessed exclusively through Edge Function helpers. Team identity uses Supabase Auth; client portal sessions are KV-backed tokens. Intelligence Gateway telemetry is in-memory only (200-record ring buffer). A legacy `DATABASE_SCHEMA.md` describes 18 relational tables that were **never deployed**.

This sprint defines the permanent multi-tenant PostgreSQL architecture, RLS model, staged KV migration, repository boundary, and bounded implementation roadmap — without modifying runtime code, applying migrations, or touching production data.

**Assumption (conservative):** MARQ Cortex operates as a single platform tenant (`MARQ` org) during early migration; multi-org tenancy is designed now but activated when product requires it.

**Assumption:** UUID v4 primary keys unless legacy KV IDs (`SUB-*`) are preserved as `legacy_id` columns during migration.

## Stage 2 — Data platform principles

| Principle | Database consequence |
|-----------|---------------------|
| Multi-tenant by design | Every tenant-owned row has `organization_id UUID NOT NULL`; composite indexes lead with `organization_id` |
| Organization ownership | FK to `organizations`; cascade rules explicit per domain |
| Strong tenant isolation | RLS on all tenant tables; `organization_id` from JWT claim or membership lookup |
| Row-level security | No direct table access from frontend; policies per role class |
| Least privilege | Separate policies for client, member, admin, service role |
| Append-only audit | `audit_events` INSERT-only for humans; no UPDATE/DELETE except legal hold tooling |
| Immutable business snapshots | `proposal_versions`, `report_versions`, `portfolio_versions` — no in-place overwrite |
| Soft deletion | `deleted_at` on user-facing entities; hard delete only for ephemeral sessions |
| Idempotent external actions | `idempotency_keys` table; email/webhook dedup |
| Clear source of truth | One authoritative store per entity; KV becomes read-only archive post-migration |
| No duplicated authority | Team roster from `memberships` + Auth; deprecate `team:member:*` KV cache |
| Deterministic business records | Scores/ROI from engines stored as typed columns + JSONB evidence |
| Provider-independent AI records | `intelligence_requests` store feature/capability; provider in protected telemetry only |
| Historical traceability | Version tables + `audit_events` + immutable snapshots |
| Migration safety | Phased dual-read/write with reconciliation reports |
| Backward compatibility | `legacy_kv_key`, `legacy_id` columns during transition |
| Data minimization | No default storage of full prompts/responses in telemetry |
| Retention/deletion | Per-table `retention_policy`; GDPR subject deletion workflow |
| Regional/privacy readiness | `organization_profiles.data_region`, export flags |
| Observable usage/costs | `usage_records`, `cost_records` with aggregation tables |
| Scalable indexing | Partial indexes on `status`, `deleted_at IS NULL` |
| Future RAG compatibility | `document_chunks` + abstracted `embeddings` store |
| Explicit lifecycle states | CHECK constraints or enum types per entity |
| Versioned schemas/records | `schema_version` on JSONB payloads; migration registry |

## Stage 4 — Relationship design (cross-cutting patterns)

### Primary key strategy

- **UUID v4** (`gen_random_uuid()`) for all new tables.
- **Legacy IDs** preserved: `submissions.legacy_id VARCHAR` for `SUB-*` keys during migration.

### Standard columns (tenant-owned tables)

```sql
id              UUID PRIMARY KEY DEFAULT gen_random_uuid()
organization_id UUID NOT NULL REFERENCES organizations(id)
created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
created_by      UUID REFERENCES auth.users(id)  -- nullable for system
updated_by      UUID REFERENCES auth.users(id)
deleted_at      TIMESTAMPTZ  -- soft delete where applicable
```

### Versioning strategy

| Entity | Strategy |
|--------|----------|
| Proposals | `proposals` + `proposal_versions` (immutable JSONB snapshot per version) |
| Reports | `reports` + `report_versions` |
| Portfolios | `portfolios` + `portfolio_versions` |
| Documents | `documents` + `document_versions` |
| Settings | Current row + `audit_events` for changes |

### Status lifecycles (examples)

- **Submission:** `new` → `in-review` → `completed` → `approved` (matches `api.ts`)
- **Proposal:** `draft` → `sent` → `viewed` → `accepted` | `rejected` | `revision_requested`
- **Intelligence request:** `queued` → `processing` → `completed` | `failed`

## Stage 5 — Multi-tenancy and RLS

### Role classes

| Role | JWT / session | Scope |
|------|---------------|-------|
| `platform_admin` | Auth claim `platform_role=admin` | All orgs (read); write via service role only |
| `org_admin` | Membership `role=admin` | Full org access |
| `team_member` | Membership `role=member` | Assigned + shared submissions |
| `team_viewer` | Membership `role=viewer` | Read-only org data |
| `client` | `client_sessions` token | Single `submission_id` scope |
| `service_role` | Edge function only | Bypass RLS for batch jobs |
| `anon` | Public | INSERT on `diagnostic_leads`, `submissions` only |

### Policy template (tenant table)

```sql
-- SELECT: org members
CREATE POLICY tenant_select ON submissions FOR SELECT
  USING (organization_id = auth.organization_id());

-- INSERT: org members + public submission endpoint via service role
CREATE POLICY tenant_insert ON submissions FOR INSERT
  WITH CHECK (organization_id = auth.organization_id());

-- UPDATE: assigned team or admin
CREATE POLICY tenant_update ON submissions FOR UPDATE
  USING (
    organization_id = auth.organization_id()
    AND (auth.is_org_admin() OR assigned_to = auth.uid())
  );

-- DELETE: soft delete only via admin
CREATE POLICY tenant_delete ON submissions FOR DELETE
  USING (organization_id = auth.organization_id() AND auth.is_org_admin());
```

### Client-scoped tables

`messages`, `engagement_events`, `proposals` (sent status), `reports` (client-visible):

```sql
CREATE POLICY client_submission_scope ON messages FOR SELECT
  USING (
    submission_id = auth.client_submission_id()
    OR organization_id = auth.organization_id()
  );
```

### Tables that must never allow direct client writes

- `submissions` (status changes), `cortex_analyses`, `usage_records`, `audit_events`, `memberships`, `organization_settings`, `intelligence_requests` (client INSERT), `pipeline_positions`

### Helper functions (justified)

- `auth.organization_id()` — resolves from JWT `app_metadata.organization_id` or membership
- `auth.is_org_admin()` — membership role check
- `auth.client_submission_id()` — from validated client session

High-volume tables and recommended indexes (see Table Catalog for full list):

| AI usage by org | `usage_records` | `(organization_id, created_at DESC)` |
| Cost by provider | `cost_records` | `(organization_id, provider_id, created_at DESC)` |
| Knowledge retrieval | `document_chunks` | `(organization_id, knowledge_space_id)` + future vector index |

| ROI actuals | `roi_actuals` | `(organization_id, submission_id, period_start)` |

**Partial indexes:** `WHERE deleted_at IS NULL`, `WHERE status = 'open'`.  
**Unique:** `(organization_id, legacy_id)`, `(organization_id, contact_email)` for active submissions.  
**GIN:** JSONB fields on `cortex_analyses.results`, `audit_events.metadata`.  
**pgvector (future):** `embeddings.embedding vector(1536)` with `ivfflat` — behind abstraction.  
**Partitioning:** `audit_events`, `usage_records` by month — only when >10M rows.

## Stage 8 — RAG and vector readiness

### Abstraction boundary

```
documents → document_versions → document_chunks → embedding_store (interface)
                                                      ├── pgvector (default recommendation)
                                                      └── external provider (pluggable)
```

### Why pgvector (recommended, requires approval)

- Co-located with tenant RLS and transactional consistency
- Simpler ops than separate vector DB for initial scale
- Replaceable via `embedding_store_provider` config

### Key tables

`knowledge_spaces`, `documents`, `document_versions`, `document_chunks`, `embeddings`, `retrieval_indexes`

### Tenant isolation

Every chunk/embed carries `organization_id` + `knowledge_space_id`; RLS inherits from space permissions.

### Deletion propagation

`ON DELETE CASCADE` from document → versions → chunks → embeddings; async re-index job marks `retrieval_indexes.stale_at`.

**Not implemented this sprint.**

## Stage 10 — AI usage and cost data

Migrate from in-memory `telemetry.ts` buffer to:

- `intelligence_requests` — request metadata, feature, org, correlation_id
- `model_attempts` — per-attempt latency, tokens, outcome
- `usage_records` — rollup per request
- `cost_records` — estimated/actual cost by provider/model

### Retention

- Raw attempts: 90 days
- Aggregated usage/cost: 2 years
- Prompts/responses: **not stored by default**; opt-in `store_content=true` per org policy

## Stage 11 — Audit and compliance

### `audit_events` (append-only)

Columns: `actor_type`, `actor_id`, `organization_id`, `action`, `entity_type`, `entity_id`, `before_state`, `after_state`, `reason`, `correlation_id`, `request_id`, `source_ip`, `environment`, `external_provider`, `result`, `failure_reason`, `occurred_at`

### Access

- Org admins: read own org audit
- Platform admins: read all
- Clients: no direct audit access

### Compliance

- Retention: 7 years default (configurable per org)
- Legal hold: `legal_holds` table flags entity immutability
- Subject deletion: anonymize `actor_id`, redact PII in `metadata`
